# 部署可行性报告（不执行迁移）

更新时间：2026-07-20。本报告只比较方案，不授权 Production 部署、环境变量修改或数据迁移。

| 方案 | 可访问性与域名/HTTPS | Node/API/Redis | 持久化、备份与日志 | 限流/隐私 | 成本与运维责任 |
|---|---|---|---|---|---|
| 医院/学校内网服务器 | 校内稳定，校外需 VPN；由机构签发域名和证书 | 同域 Node 22 服务；机构 Redis | 机构备份策略、集中日志、灾备 | 最利于数据边界；需院方安全审查与网关限流 | 硬件/机房已有时边际成本低；IT 部门承担 7×24 运维 |
| 阿里云 | 国内公网可达性较好；备案域名与 HTTPS 必需 | ECS/容器/函数 + 云 Redis | 快照、跨区备份、日志服务 | WAF、限流、等保/隐私合规由项目方配置 | 按量或包年；项目方承担云账号、补丁和告警 |
| 腾讯云 | 国内公网可达性较好；备案和证书必需 | CVM/容器/云函数 + Redis | 云备份与日志服务 | WAF、限流、数据地域需明确 | 与阿里云相近；需指定长期运维负责人 |
| 华为云 | 政企/医疗采购渠道可能更合适；备案和证书必需 | ECS/CCE/函数 + DCS Redis | 云备份、LTS 日志 | 安全产品齐全，仍需机构制度落地 | 采购与架构复杂度较高；适合已有华为云体系的机构 |
| 现有 Vercel 全栈 | 海外边缘网络；教师网络需实测；Preview 可受保护 | Next/Vercel Functions + Marketplace Upstash/KV REST | Redis 持久；部署回滚便捷；函数日志需脱敏 | 同源 API 最简单；Preview/Production 环境严格隔离 | 初期运维低；受套餐、函数时限和外部服务可达性约束 |
| GitHub Pages + 独立 API | 静态页面易发布，但国内与跨域体验不确定 | Pages 无 Node；API/Redis 必须另建 | 前端无状态，后端单独备份/日志 | 必须精确 CORS；保护页可能阻断 OPTIONS；攻击面更复杂 | 静态成本低但双平台运维成本高，不建议作为完整 AI 训练主入口 |

## 推荐目标架构

若需要面向国内教师稳定开放，优先选择“机构内网同域部署”或“国内云同域全栈”：前端、API、持久化 attempt store 和 AI 代理处于同一受控域名，使用 HTTPS、Node 22、可写 Redis、环境变量密钥管理、自动备份、脱敏日志、速率限制和明确值班责任。不得让教师访问开发者电脑的 localhost。

短期继续以受保护 Vercel Preview 做审阅；GitHub Pages 只作为静态展示或清晰引导到授权完整站点。正式迁移前必须另行批准域名/备案、费用、账号、数据地域、备份恢复演练、隐私影响评估和 Production 切换/回滚计划。
