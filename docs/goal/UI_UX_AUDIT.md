# UI/UX 基线审计

审计日期：2026-07-13  
实际基线：`74c140fb77844ee557c739112c68076113375e25`  
基线分支：远程 `codex/hematuria-production-goal`  
UI 专项分支：`codex/hematuria-ui-polish`

## 审计范围与方法

- 使用真实 Chromium 页面与 Playwright DOM/交互接口检查 `1440×900`、`1280×720`、`390×844`、`360×800`。
- 检查中文、英文、键盘发送、规则库降级、AI 连接失败、日志等待同步、空状态与控制台。
- 核对 `app/globals.css`、Tailwind 配置、首页、病例目录、七阶段训练工作台、报告卡和 360 分报告组件。
- 当前版本只有浅色主题（根节点声明 `color-scheme: light`，无深色 Token 或切换入口），因此深色模式记录为“未支持”，不是本轮回归基线。
- 本地前端默认依赖独立训练 API（`http://127.0.0.1:3001`）。仓库未提供完整本地训练 API 启动器，故真实页面可审计第 1 阶段与降级路径，但不能从真实服务推进到第 2–7 阶段；这些面板以源码审计、既有拦截式 E2E 和本轮结构测试补足。该问题属于工程环境缺陷，不在 UI 任务中重构。

## 核心结论

1. 问诊页功能入口完整，但桌面信息密度偏高；`1280×720` 与移动端首屏被标题、模式开关和状态控制占用，输入区远离对话上下文。
2. AI 状态同时出现在来源标签、连接提示、健康提示和日志同步横幅中，降级时可堆叠两至三个黄色/蓝色提示，并推动主布局下移。
3. 聊天输入是单行 `input`，不支持 Shift+Enter 多行；移动端输入区在约 `y=1087`，不稳定可见。
4. 病例目录卡片清楚且不泄露诊断，但缺搜索、主题入口、进度/继续训练状态和自然的无结果空状态。
5. 现有 Token 数量少且直接颜色值散落；没有暗色 Token、统一按钮/卡片/状态组件、减弱动画规则或统一触控尺寸。
6. 训练数据面板使用一致卡片框架，但异常、待审核、正常等语义主要依赖自由文本和颜色；长报告缺少明确的值/单位/参考范围结构。
7. 360 分结果与实际分项数据绑定，未使用装饰图表；但总分、强项、改进项和危险遗漏仍挤在同一密度层级，打印样式只做了基础隐藏。

## 可访问性与性能基线

- 优点：主导航有名称与当前页标识；核心表单多有可访问名称；焦点轮廓已定义；消息区使用 `role=log`；弹窗标记为 dialog。
- 问题：多个 28–38px 控件低于推荐移动触控高度；弹窗未见 Escape 关闭逻辑；Header/顶栏在移动端占位过大；持续 `animate-pulse` 用于报告等待且未尊重 `prefers-reduced-motion`；根元素语言不会随界面语言更新；无暗色对比度策略。
- 基线无新大型依赖。视觉修改必须继续使用 Tailwind/CSS 与现有 `lucide-react`，不引入 UI 框架。

## 不属于本轮 UI 的工程缺陷

1. 仓库缺少可一键启动的完整本地训练 API；`dev:api` 仅启动 Patient Agent 的 `8787` 接口，而前端默认请求 `3001` 的 session/training/health 等端点。
2. 本地开发 API 不可用时，阶段 1 提交无法推进第 2–7 阶段；生产与 CI 的拦截/服务环境另有覆盖。本轮只记录，不重构核心状态机或 API。
